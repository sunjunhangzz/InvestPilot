# 00-代码规范总则

## 基本原则

代码必须服务于第一版目标：

```text
稳定生成 A股主板稳健趋势股推荐结果
```

禁止为了未来功能提前引入复杂抽象。

## 可读性

代码优先保证：

- 名称清晰
- 函数短小
- 边界明确
- 错误可追踪
- 数据可复盘

不要写“聪明但难懂”的代码。

## 单一职责

每个模块只做一类事情：

- `web` 展示和 API
- `worker` 数据和计算
- `shared` 配置和约定

不要在前端里写因子计算，不要在 worker 里写页面逻辑。

## 命名

统一约定：

```text
Python 文件、函数、变量：snake_case
SQLite 表和字段：snake_case
TypeScript 变量和函数：camelCase
React 组件：PascalCase
常量：UPPER_SNAKE_CASE 或清晰的 const camelCase
```

## 时间字段

必须使用明确字段：

```text
trade_date
created_at
updated_at
started_at
finished_at
```

所有推荐结果必须关联：

```text
run_id
```

所有任务必须关联：

```text
task_id
```

## 错误处理

禁止：

- 静默失败
- 只打印错误但不记录状态
- 捕获异常后继续写错误数据

必须：

- 任务失败写 `system_tasks`
- 详细错误写日志
- API 返回明确错误
- 前端展示错误摘要

## 注释

注释只解释：

- 为什么这样做
- 业务规则
- 风险边界
- 外部接口特殊处理

不要注释显而易见的代码。

## 注释同步

代码变更时必须同步检查注释。

以下场景必须写注释：

- 策略筛选规则有业务含义
- 因子计算存在边界处理
- 为避免未来函数而做特殊处理
- 为兼容 AkShare 字段变化而做适配
- 为避免 SQLite 写锁而做串行控制
- AI 输入被压缩或裁剪
- 错误处理不是直观流程

以下场景不要写注释：

- 变量名和函数名已经表达清楚
- 注释只是重复代码含义
- 临时想法但没有实际约束

禁止保留过期注释。

如果代码逻辑变化导致注释不再准确，必须在同一次修改中更新注释。
