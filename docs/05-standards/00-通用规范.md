# 00-通用规范

## 项目边界

第一版只做投资研究辅助。

禁止：

- 自动交易
- 自动下单
- 连接券商账户
- AI 直接决定买卖
- 实验模块直接影响正式推荐

## 开发原则

- 不提前引入复杂架构。
- 不为未来功能牺牲第一版稳定性。
- 不在当前任务中实现后续阶段能力。
- 代码和文档冲突时，先更新文档再实现。
- 每个任务都要能独立验证。

## 目录原则

```text
app/web       前端页面和 API Routes
app/worker    Python 数据和计算层
app/shared    公共配置和字段约定
app/api       后续独立后端预留，第一版不开发
data          数据库、缓存、日志
reports       报告输出
docs          文档
backups       备份
```

不要把代码、数据、报告、文档混放。

## 命名原则

文件名：

- Markdown 文档可以使用中文。
- 代码文件使用英文小写和下划线或短横线。
- Python 文件使用 snake_case。
- TypeScript 组件使用 PascalCase。

变量名：

- Python 使用 snake_case。
- 数据库字段使用 snake_case。
- TypeScript 使用 camelCase。
- React 组件使用 PascalCase。
- 常量使用 UPPER_SNAKE_CASE 或清晰的 camelCase 常量。

## 时间和日期

- 所有交易数据必须带 `trade_date`。
- 所有任务必须带 `task_id`。
- 所有推荐运行必须带 `run_id`。
- 时间字段统一保存 ISO 格式字符串。
- 展示时再转换为本地时间。

## 错误处理

- 不吞错误。
- 不只打印错误。
- 任务失败必须写入 `system_tasks`。
- 详细错误必须写入 `data/logs/`。
- 前端必须显示可理解的错误摘要。

## 注释原则

注释必须及时写入，并和代码同步更新。

注释只解释：

- 为什么这样做
- 业务规则
- 风险边界
- 外部接口特殊处理
- 非直观的错误处理

不要注释显而易见的代码。

当修改以下内容时，必须检查并更新相关注释：

- 筛选规则
- 因子计算逻辑
- 任务状态流转
- 数据库写入规则
- AI 输入输出约束
- 风险过滤规则
- 外部数据源适配

过期注释比没有注释更危险，发现过期注释必须立即删除或修正。
