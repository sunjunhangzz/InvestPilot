# 日志目录约定

## 目标

统一第一版日志文件位置和用途，让数据采集、筛选、AI 和 API 失败后都能定位问题。

日志分两层：

```text
system_tasks      任务摘要，供前端展示
data/logs/*.log   详细日志，供本地排查
```

## 日志目录

本地详细日志统一写入：

```text
data/logs/
```

`data/logs/` 只提交 `.gitkeep`，真实日志文件不进入 Git。

## 日志文件

第一版固定使用这些文件：

```text
data/logs/worker.log       worker 通用任务日志
data/logs/data_source.log  AkShare 和外部数据源日志
data/logs/screening.log    因子计算和推荐筛选日志
data/logs/ai.log           AI 报告调用日志
data/logs/api.log          Next.js API Routes 日志
```

## 写入职责

worker 写：

```text
worker.log
data_source.log
screening.log
ai.log
```

web API Routes 写：

```text
api.log
```

数据库任务摘要写：

```text
system_tasks.error_message
```

## 任务失败记录规则

任务失败时必须同时记录：

- `system_tasks.status = failed`
- `system_tasks.error_message` 写短错误摘要
- `data/logs/*.log` 写详细错误、上下文和堆栈

`error_message` 面向前端展示，只放可以给用户看的摘要。

详细堆栈只写入本地日志，避免前端暴露内部路径、环境变量或第三方接口细节。

worker 任务失败统一使用：

```text
mark_task_failed_with_exception()
```

该函数负责：

- 把详细异常写入 `data/logs/*.log`
- 把短摘要写入 `system_tasks.error_message`
- 对过长摘要进行裁剪

## 数据采集失败

数据采集失败时至少记录：

- `task_id`
- `run_id`
- 数据源名称
- 股票代码，批量失败时记录失败数量和样例
- 失败原因
- 是否保留旧数据

数据源返回空数据时必须记录日志，并且不能覆盖已有可用数据。

## AI 调用失败

AI 调用失败时至少记录：

- `task_id`
- `run_id`
- provider
- model
- report_type
- 失败原因

禁止记录 API Key、完整请求头、完整 `.env`。

AI 失败不能影响基础推荐结果。

## 日志格式

统一使用 JSON Lines：

```json
{"time":"2026-07-04T20:00:00+08:00","level":"INFO","module":"update_prices","task_id":"task_001","run_id":"run_001","message":"start update prices","context":{"stock_count":1200}}
```

必须字段：

```text
time
level
module
message
```

任务相关日志必须带：

```text
task_id
run_id
```

交易日相关日志必须带：

```text
trade_date
```

## 统一工具

worker 统一使用：

```text
app/worker/src/loggers/json_logger.py
```

web API Routes 统一使用：

```text
app/web/lib/server/logger.ts
```

不要在业务代码里手写不同格式的日志落盘逻辑。

## 保留策略

第一版先不实现自动清理。

后续日志工具实现时按以下目标处理：

- 日志按文件大小或日期滚动
- 默认保留最近 30 天
- 本地大日志不提交到 Git
