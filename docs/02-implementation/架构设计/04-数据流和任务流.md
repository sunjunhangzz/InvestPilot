# 04-数据流和任务流

## 数据流

```text
AkShare
  ↓
worker/update_stocks.py
  ↓
stocks

AkShare
  ↓
worker/update_prices.py
  ↓
daily_prices

daily_prices + stocks
  ↓
worker/calc_factors.py
  ↓
factors

factors + stocks
  ↓
worker/run_screening.py
  ↓
recommendations

recommendations + daily_prices
  ↓
worker/update_watchlist.py
  ↓
watchlist

recommendations + factors
  ↓
worker/generate_report.py
  ↓
ai_reports

SQLite
  ↓
web API Routes
  ↓
前端页面
```

## 任务流

用户点击按钮后：

```text
前端
  ↓
POST /api/tasks/update-data
  ↓
API 创建 task_id
  ↓
API 写入 system_tasks pending
  ↓
API 启动 worker 子进程
  ↓
worker 写入 system_tasks running
  ↓
worker 执行并写数据库
  ↓
worker 写入 system_tasks success/failed
  ↓
worker 写入 data/logs 详细日志
  ↓
前端轮询 GET /api/tasks/:task_id
  ↓
前端刷新结果
```

## run_id

`run_id` 用于标识一次完整推荐运行。

必须绑定：

- factors
- recommendations
- ai_reports

默认展示最近一次成功 `run_id`。

## task_id

`task_id` 用于标识一个执行任务。

例如：

- update-data
- run-screening
- generate-report

一个 `run_id` 可以对应多个 `task_id`。

## 任务状态写入职责

```text
API Routes
  负责创建 task_id
  负责写入 pending
  负责启动 worker 子进程

worker
  负责写入 running
  负责写入 success 或 failed
  负责写入 error_message 摘要
```

如果 worker 异常退出且没有更新状态，API 需要根据子进程退出码把任务标记为 failed。

## 串行写入

第一版所有写数据库的 worker 任务串行执行。

禁止：

- 两个 update_prices 同时运行
- update_prices 和 run_screening 同时运行
- 两个 run_screening 同时运行

原因：

- 避免 SQLite 写锁
- 避免数据日期不一致
- 避免推荐结果混乱

## 失败处理

失败时：

- worker 返回非 0 exit code
- system_tasks 写 failed
- error_message 写摘要
- 日志写详细错误
- 前端显示失败原因
- 允许用户重试

AI 失败不影响推荐结果。

数据源返回空数据不覆盖旧数据。
