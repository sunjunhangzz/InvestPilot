# 日志体系

## 目标

日志体系用于定位问题、追踪任务执行、复盘数据质量和排查 AI 调用失败。

第一版日志目标：

- 能知道任务什么时候开始、什么时候结束。
- 能知道任务失败在哪一步。
- 能知道数据采集失败的股票代码和原因。
- 能知道 AI 调用是否成功、失败原因是什么。
- 能在网页或数据库中看到任务摘要。
- 能在本地日志文件中看到详细错误。

## 日志分层

第一版采用两层日志：

```text
数据库任务日志：system_tasks
本地文件日志：data/logs/
```

## 数据库任务日志

`system_tasks` 记录任务级状态。

适合展示在前端：

- task_id
- run_id
- task_name
- status
- started_at
- finished_at
- error_message

用途：

- 前端展示任务状态
- 判断任务是否失败
- 支持失败重试
- 查看最近一次任务摘要

## 本地文件日志

详细日志写入：

```text
data/logs/
```

建议文件：

```text
worker.log
data_source.log
screening.log
ai.log
api.log
```

用途：

- 记录详细错误堆栈
- 记录失败股票列表
- 记录外部接口异常
- 记录 AI 调用失败
- 记录任务执行耗时

## 日志格式

第一版建议使用 JSON Lines。

每行一条日志：

```json
{
  "time": "2026-07-04T09:00:00+08:00",
  "level": "INFO",
  "module": "update_prices",
  "task_id": "task_20260704_001",
  "run_id": "20260704-manual-001",
  "message": "start update prices",
  "context": {
    "stock_count": 1200
  }
}
```

日志等级：

```text
DEBUG
INFO
WARN
ERROR
```

## 日志收集

第一版不引入外部日志系统。

日志收集方式：

```text
worker 写 data/logs/*.log
API 写 data/logs/api.log
system_tasks 保存任务摘要
前端通过 API 读取最近任务状态
```

后续如果部署到服务器，可扩展到：

- Loki
- ELK
- OpenTelemetry
- 云日志服务

第一版暂不做这些。

## 日志保留

第一版建议：

- 日志文件按大小或日期滚动。
- 保留最近 30 天日志。
- 大日志不进入 Git。

`.gitignore` 必须排除：

```text
data/logs/
```

## 前端展示

第一版前端只展示任务摘要，不展示完整日志。

展示内容：

- 最近任务状态
- 失败原因摘要
- 任务开始时间
- 任务结束时间

后续可以增加：

- 日志查看页面
- 失败股票列表
- 数据源健康状态

## 必须记录的事件

数据采集：

- 开始采集
- 采集完成
- 股票列表数量
- 行情成功数量
- 行情失败数量
- 失败股票代码和原因
- 数据源返回空数据

因子和推荐：

- run_id
- 参与计算股票数量
- 被过滤股票数量
- 推荐股票数量
- A/B/C 分层数量

观察池：

- 新增观察股票
- 更新观察股票
- 降级股票
- 退出股票
- 退出原因

AI：

- AI 是否启用
- 模型名称
- 调用成功数量
- 调用失败数量
- 单只股票失败原因
- token 使用量，后续加入

API：

- 创建 task_id
- 拒绝重复任务
- 子进程启动
- 子进程退出码
- API 错误
