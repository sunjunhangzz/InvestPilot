# 第一版实现方案

## 实现原则

第一版采用逻辑分层、物理简单的结构。

```text
web      前端页面和 Next.js API Routes
worker   Python 数据采集、因子计算、推荐生成
shared   公共配置、字段约定、运行参数
data     SQLite 数据库和缓存
reports  报告输出
```

第一版不拆微服务，不引入消息队列，不使用 PostgreSQL、Redis、Docker。

## 子项目划分

```text
app/
  web/       Next.js 网站，包含前端页面和 API Routes
  worker/    Python 脚本，负责数据、计算、推荐、报告
  shared/    共享配置、Schema、运行约定
  api/       预留目录，后续独立后端时使用
```

## 核心数据流

```text
用户点击按钮
  ↓
Next.js API 创建任务
  ↓
写入 system_tasks，返回 task_id
  ↓
后台调用 Python worker
  ↓
worker 写入 SQLite
  ↓
前端轮询任务状态
  ↓
前端读取推荐、观察池和报告
```

## 第一版任务顺序

任务必须串行执行，避免 SQLite 写锁和数据不一致。

推荐执行顺序：

```text
1. init_db
2. update_stocks
3. update_prices
4. calc_factors
5. run_screening
6. update_watchlist
7. generate_report
```

## 关键工程约束

- 所有任务必须有 `task_id`。
- 每次推荐运行必须有 `run_id`。
- 所有推荐结果必须带 `trade_date` 和 `run_id`。
- 同一时间只允许一个写入任务运行。
- AI 失败不能影响基础推荐列表。
- 数据源失败不能覆盖已有可用数据。
- 重复运行不能直接覆盖历史结果。
- 非交易日运行时应提示，并默认使用最近一个交易日数据。
- 前端长任务必须显示运行状态，不能一直卡住。

## 推荐运行标识

`run_id` 用来区分同一天多次运行。

示例：

```text
20260704-manual-001
20260704-morning-001
20260704-afternoon-001
```

第一版默认展示最近一次成功运行的推荐结果。

## 数据原则

- 第一版使用前复权日线数据计算均线和收益率。
- 推荐基于最近可用交易日。
- 过滤 ST、退市风险、停牌、上市不足 180 天、低流动性股票。
- 涨停、跌停股票第一版先打标签，不一定完全剔除。
- 数据缺失股票跳过，并记录日志。

## AI原则

- AI 只生成解释和风险提示。
- AI 不直接决定推荐列表。
- 没有 API Key 时，系统仍可输出基础推荐。
- AI 调用失败时，记录错误并允许重试。
