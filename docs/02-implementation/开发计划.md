# 开发计划

## 统筹原则

第一版按主链路依赖顺序开发：

```text
shared配置 → 数据库 → worker数据层 → worker筛选层 → 观察池 → API任务层 → 前端展示 → AI报告 → 联调
```

核心原则：

- 先 worker，后 web。
- 先数据，后页面。
- 先规则，后 AI。
- 先手动，后自动。
- 先稳定，后复杂。

第一版实际开发 3 个代码子项目：

```text
app/shared   公共配置、字段约定、路径约定
app/worker   Python 数据采集、因子计算、推荐生成
app/web      Next.js 页面和 API Routes
```

`app/api` 是后续独立后端预留目录，第一版不开发。

## 第0阶段：项目基建

目标：统一项目约定，让后续代码不会各写各的路径和字段。

开发内容：

- 检查 `app/shared/config.json`
- 检查 `app/shared/schema.json`
- 创建 Python 虚拟环境
- 创建 Next.js 项目
- 创建 `.env` 配置文件
- 创建 `.gitignore`

完成标准：

- `python3` 可运行。
- `npm` 可运行。
- TS 和 Python 都能读取 shared 配置。
- 数据库路径、报告路径、推荐数量、因子权重有统一配置。

风险点：

- 配置路径相对位置容易写错。
- TS 使用 camelCase，Python/SQLite 使用 snake_case，需要集中转换。

## 第1阶段：数据库

目标：建立稳定的数据底座。

开发内容：

```text
worker/scripts/init_db.py
```

创建表：

- runs
- system_tasks
- stocks
- daily_prices
- factors
- recommendations
- watchlist
- ai_reports
- settings

完成标准：

- 能生成 `data/a_stock_research.sqlite`。
- 重复运行 `init_db.py` 不报错。
- 表和索引都创建成功。
- 能写入一条测试任务记录。

风险点：

- SQLite 写锁。
- 表结构和文档不一致。
- 重复初始化误删数据。

约束：

- 初始化只创建缺失表，不删除已有数据。
- 写入必须使用事务。

## 第2阶段：数据采集

目标：拿到 A股主板股票和最近日线行情。

开发内容：

```text
worker/scripts/update_stocks.py
worker/scripts/update_prices.py
```

第一版采集范围：

- A股股票列表
- 主板股票池
- 最近约 120 个交易日日线
- 前复权行情优先

完成标准：

- `stocks` 有主板股票数据。
- `daily_prices` 有最近日线数据。
- 数据源返回空数据时不覆盖旧数据。
- 缺失数据能记录到任务错误或日志。

风险点：

- AkShare 接口字段变化。
- 网络失败。
- 非交易日没有当天数据。
- 停牌、ST、主板判断字段不稳定。

约束：

- 推荐基于最近可用交易日。
- 非交易日默认使用最近一个交易日数据。

## 第3阶段：因子和推荐

目标：从主板股票池中筛出 10-50 只候选股票。

开发内容：

```text
worker/scripts/calc_factors.py
worker/scripts/run_screening.py
```

第一版因子：

- MA20
- MA60
- 20日收益率
- 60日收益率
- 近20日日均成交额
- 近20日日均换手率
- 波动率
- 最大回撤
- 总评分

完成标准：

- 每次运行生成 `run_id`。
- `factors` 有评分数据。
- `recommendations` 有推荐结果。
- 推荐数量控制在 10-50 只。
- A类、B类、C类分层可用。

风险点：

- 未来函数。
- 复权数据不一致。
- 缺少 60 日数据导致均线为空。
- 低流动性股票混入推荐。

约束：

- 不使用未来数据。
- 缺少足够历史数据的股票跳过。
- 第一版不依赖财报、新闻、情绪数据。

## 第4阶段：观察池

目标：保证推荐连续性。

开发内容：

```text
worker/scripts/update_watchlist.py
```

规则：

- 推荐股票进入观察池。
- 最少跟踪 5 个交易日。
- 重复推荐不覆盖首次推荐日期。
- 退出观察必须记录原因。

完成标准：

- `watchlist` 有数据。
- 已跟踪天数能更新。
- 状态能在 `active / hold / downgraded / exit / blocked` 之间流转。
- 推荐股票不会无原因消失。

风险点：

- 交易日计数和自然日计数混淆。
- 同一股票重复插入。
- 退出逻辑过于激进。

约束：

- 第一版按交易日计算跟踪周期。
- 不因单日评分下降立即移出。

## 第5阶段：API任务层

目标：让网页能安全触发 worker，并读取结果。

开发内容位于 `app/web` 的 Next.js API Routes：

```text
POST /api/tasks/run (body: {"pipeline": "update-data"})
POST /api/tasks/run (body: {"pipeline": "run-screening"})
POST /api/tasks/run (body: {"pipeline": "generate-report"})
GET  /api/tasks/:task_id
GET  /api/runs/latest
GET /api/recommendations
GET  /api/stocks/:code
GET  /api/watchlist
GET  /api/settings
POST /api/settings
```

完成标准：

- 长任务接口立即返回 `task_id`。
- 前端可以查询任务状态。
- API 默认读取最近一次成功 `run_id`。
- Python 执行失败时，API 能返回错误状态。

风险点：

- API 等待 Python 完成导致超时。
- 重复点击导致多个写任务并发。
- Python 子进程退出码未正确处理。

约束：

- 写入任务串行执行。
- 同一时间只允许一个 worker 写数据库。
- AI 失败不能影响推荐接口。

## 第6阶段：前端页面

目标：用户能在本地网页查看结果。

开发内容：

- 首页仪表盘
- 今日推荐
- 股票详情
- 观察池
- 系统设置

完成标准：

- 能显示最新成功 `run_id`。
- 能查看推荐列表。
- 能查看股票详情。
- 能查看观察池状态。
- 数据为空、任务失败、AI 未配置都有提示。

风险点：

- 推荐表格字段过多。
- 长任务没有进度反馈。
- 股票详情一次加载过多行情导致页面慢。

约束：

- 列表只展示核心字段。
- 详情页展示完整信息。
- 所有页面必须有 loading / empty / error 状态。

## 第7阶段：AI基础报告

目标：AI 只做解释和排雷，不影响基础推荐。

开发内容：

```text
worker/scripts/generate_report.py
```

页面展示：

- 股票详情页展示 AI 解释。
- 首页或推荐页展示 AI 是否生成成功。

完成标准：

- 没有 API Key 时系统仍可用。
- AI 调用失败时推荐列表仍正常。
- 有 API Key 时能生成简版个股解释报告。
- AI 报告写入 `ai_reports`。

风险点：

- API Key 未配置。
- API 调用超时。
- 返回内容格式不稳定。
- token 成本失控。

约束：

- 第一版不做真正多 Agent。
- 只对最终推荐股票生成报告。
- 不把原始大段行情数据直接传给模型。

## 第8阶段：本地联调

目标：完整跑通第一版日常使用流程。

完整流程：

```text
初始化数据库
更新股票
更新行情
计算因子
生成推荐
更新观察池
生成报告
网页查看
```

完成标准：

- 一组按钮可以跑通主流程。
- 失败时能看到错误原因。
- 重复运行不会破坏旧数据。
- 非交易日能使用最近交易日数据。
- AI 失败不影响推荐。
